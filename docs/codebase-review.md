# ZombieForge Codebase Review

> Generated: 2026-04-18. Covers `ZombieForge/` (WinUI 3 / .NET 8 / C# 12) and `BlackOpsMonitor/` (native C++ DLL injected into BO1). Autogenerated files under `obj/`, `bin/`, `Debug/`, `.vs/` and Assets were skipped.

---

## Executive Summary

ZombieForge is a small, cleanly-structured single-window WinUI 3 app with a reasonably disciplined MVVM split, structured logging via Serilog, and a clear path for adding new games via `IGameHandler`. For a project of this size the architecture is sound, naming is consistent, nullable reference types are enabled, and the per-game abstraction is honored. The biggest risks lie below the UI layer: cross-architecture DLL injection is not validated, the native hook is version-locked to a single BO1 build with no verification (so a wrong-patch user will crash the game), several native-memory P/Invoke paths silently swallow failures and return 0, and there is no automated test coverage at all.

The most impactful improvements would be (1) validating target-process bitness and game-version before hooking/injection, (2) making `MemoryService.ReadInt32` distinguish failure from a real zero, (3) tightening `GameEventMonitor` shutdown so `Dispose` cannot block the UI thread for up to 5 seconds, and (4) fixing the trimming configuration which is likely to break XAML at runtime in Release builds.

## Critical Issues

### C-1 · DLL injection does not validate target architecture
- **File:** `ZombieForge/Services/DllInjector.cs:48-125`, `ZombieForge/ZombieForge.csproj:8-9`
- **Severity:** critical · **Category:** Security / Correctness
- BlackOps.exe is a 32-bit process, but `ZombieForge` builds for `x86;x64;ARM64`. `DllInjector` resolves `LoadLibraryW` via `GetProcAddress(GetModuleHandleA("kernel32.dll"), ...)` in the *caller's* address space and then passes that pointer to `CreateRemoteThread` in the *target*. When ZombieForge is launched as x64 or ARM64 and the target is 32-bit, the pointer is invalid in the remote process and `BlackOpsMonitor.dll` cannot be loaded into the 32-bit target anyway (bitness must match). There is no check that target PID is WOW64 / matches the current process architecture, and no user-facing error — it just silently fails.
- **Fix:** Call `IsWow64Process2` on both processes before injection; abort with a clear log/UI message if they mismatch. Ship only `win-x86` for the injection path, or use a 32-bit helper EXE. Example:
  ```csharp
  if (!ArchitectureMatches(Process.GetCurrentProcess(), targetProc))
  {
      logger.LogWarning("Architecture mismatch: cannot inject into {Pid}", targetProc.Id);
      return false;
  }
  ```

### C-2 · Native hook has no game-version verification
- **File:** `BlackOpsMonitor/Hook.cpp:7-17`, `ZombieForge/Services/Games/BO1AddressProfiles.cs:10-20`
- **Severity:** critical · **Category:** Security / Correctness
- `VM_NOTIFY_ADDR = 0x008A87C0`, `SL_CONVERT_TO_STRING = 0x00687530`, `LEVEL_TIME_ADDR = 0x0286D014`, and the 6-byte stolen prologue are hard-coded for one specific build (`1.3.0.780`). If a user is on any other patch or a modded executable, `InstallHook` overwrites six bytes of arbitrary code and the game crashes on the first script notification. `BO1AddressProfiles` stores a `Version` string but it is never compared to the actual module's checksum/timestamp.
- **Fix:** Before `InstallHook`, read the PE header (`NT_HEADERS->FileHeader.TimeDateStamp` or ImageSize) from the loaded `BlackOps.exe` and bail out if it does not match a known profile; surface the result back to the managed side via `SharedGameState` so the UI can show "unsupported game version".

### C-3 · `MemoryService.ReadInt32` silently returns 0 on failure
- **File:** `ZombieForge/Services/MemoryService.cs:25-38`
- **Severity:** critical · **Category:** Error Handling / Correctness
- `ReadInt32` calls `TryReadInt32`, discards the success flag and returns `BitConverter.ToInt32(buffer, 0)`. On any failure the caller sees `0`, which is indistinguishable from a legitimate zero (e.g. start of a round, no kills). `BlackOps1Handler.ReadPlayerStats` and `ReadLevelTime` both use this overload, so the UI will display "0 points / 0 kills" whenever the read fails transiently (handle closed by us elsewhere, process exiting, paged-out memory, etc.) — creating data-loss-feeling UX and potentially poisoning `GameSession` timing logic (which treats `currentMs <= 0` as "not active" and resets state).
- **Fix:** Remove the lossy overload and make callers use `TryReadInt32`, propagating failure as nullable or an explicit `bool`:
  ```csharp
  if (!MemoryService.TryReadInt32(handle, addr, out var value, out _)) return PlayerStats.Unknown;
  ```

### C-4 · `PublishTrimmed=True` will break WinUI XAML in Release
- **File:** `ZombieForge/ZombieForge.csproj:60-61`
- **Severity:** critical · **Category:** Dependency Health / Correctness
- `PublishTrimmed` is enabled for every non-Debug configuration. WinUI 3 relies heavily on reflection (generated `XamlTypeInfo`, converters resolved by type name, data templates resolved by `x:DataType`). Trimming without `TrimmerRootAssembly`/`TrimmerRootDescriptor` entries will strip types used only from XAML, producing `XamlParseException`/`COMException` at first navigation in Release builds. WindowsAppSDK itself is not certified trim-safe.
- **Fix:** Set `<PublishTrimmed>false</PublishTrimmed>` unconditionally, or opt-in per-assembly with explicit roots. Also consider turning off `PublishReadyToRun` until you actually need it; the defaults are fine.

### C-5 · WMI query is built with string interpolation
- **File:** `ZombieForge/Services/ProcessWatcher.cs:27-35`
- **Severity:** critical · **Category:** Security (defense-in-depth)
- The executable name is hard-coded today, but `ProcessWatcher.Watch` takes `IEnumerable<string>`, and the WQL text is assembled with `$"... TargetInstance.Name = '{exe}'"`. If a future caller ever passes a user-provided string (e.g. from a config file), a single quote injects arbitrary WQL. The class should whitelist or escape input so this cannot regress.
- **Fix:** Validate `name` against `^[A-Za-z0-9_.\- ]+$` before building the query, or reject any input containing `'`.

### C-6 · `GameEventMonitor.Dispose` can block the UI thread up to 5 s
- **File:** `ZombieForge/Services/GameEventMonitor.cs:57-60, 116-123`
- **Severity:** critical · **Category:** Performance / Correctness
- `RunAsync` uses `Task.Run(() => _event!.WaitOne(TimeSpan.FromSeconds(5)), ct)`. The inner `WaitOne` is **not cancellable** — cancelling `ct` only prevents the next iteration; the current wait must time out first. `Dispose` then calls `_runTask.Wait()` (synchronous, unbounded). `Dispose` is invoked from `MainViewModel.Dispose` which is wired to `MainWindow.Closed`, so closing the app hangs the UI message pump for up to 5 s every time.
- **Fix:** Use `WaitHandle.WaitAny(new[] { _event, ct.WaitHandle }, 5000)` so the wait unblocks on cancellation, or register a callback that calls `_event.Set()` on cancel. Also make `Dispose` `await`-based (`DisposeAsync`) and avoid `.Wait()` on UI thread.

### C-7 · Shared-memory event protocol drops events under load
- **File:** `BlackOpsMonitor/Hook.cpp:78-91`, `ZombieForge/Services/GameEventMonitor.cs:57-70`
- **Severity:** critical · **Category:** Correctness
- The protocol is a single slot (`lastEvent`, `eventTimestamp`, `eventValue`). If two script events fire before C# drains (common — `end_of_round` and `start_of_round` can be microseconds apart, and `powerup_grabbed` can burst), the second write overwrites the first and `SetEvent` is called twice on an auto-reset event, so C# sees only one. Events are silently lost.
- **Fix:** Replace the single slot with a ring buffer (head/tail indices, N×`EventRecord` slots) and have C# drain all pending records on each signal.

### C-8 · Requiring admin in an MSIX-packaged app will fail to launch
- **File:** `ZombieForge/app.manifest:5-11`, `ZombieForge/Package.appxmanifest:49-52`
- **Severity:** critical · **Category:** Architecture / Correctness
- `requestedExecutionLevel level="requireAdministrator"` in `app.manifest` is *ignored* by MSIX and in some scenarios actively breaks launch. MSIX apps run with the user's token; elevation is not supported through `runFullTrust` alone. If a user installs the MSIX and double-clicks the tile, the app starts un-elevated and every `OpenProcess(... VM_WRITE ...)` against `BlackOps.exe` (which runs as a normal user too, so this *may* work, depending on UIPI/integrity level) will silently fail when the user is on a locked-down system or the game ever launches elevated.
- **Fix:** Either ship as an **unpackaged** MSIX-free EXE that *can* honour the manifest's `requireAdministrator`, or drop the admin requirement and document that BO1 must not be started as admin. Pick one delivery model consistently.

## Major Issues

### M-1 · `MainViewModel` exposes UI types (architectural rule violation)
- **File:** `ZombieForge/ViewModels/MainViewModel.cs:81-83`, `ZombieForge/Converters/BoolToColorConverter.cs`
- **Severity:** major · **Category:** Architecture
- Project instructions ("ViewModels expose data to the UI — no UI types in services or models") are contradicted by `StatusBrush` returning a `SolidColorBrush`. Worse, a new brush is allocated on every property access (and every `OnPropertyChanged(nameof(StatusBrush))`). There is already a `BoolToColorConverter` intended for exactly this — but it is dead code. Bind `IsGameRunning` and let the converter produce the brush.
- **Fix:** Remove `StatusBrush`; in XAML: `Fill="{x:Bind ViewModel.IsGameRunning, Mode=OneWay, Converter={StaticResource BoolToColor}}"`.

### M-2 · Re-injection on app restart will double-hook BO1
- **File:** `ZombieForge/ViewModels/MainViewModel.cs:137-158`, `BlackOpsMonitor/dllmain.cpp:88-106`
- **Severity:** major · **Category:** Correctness
- `BlackOpsMonitor.dll` is injected on every start. If the user closes ZombieForge (which does *not* eject the DLL from BO1 — the DLL stays loaded because `FreeLibrary` is never called remotely) and then reopens ZombieForge while BO1 is still running, `LoadLibraryW` on a DLL that is already loaded is a no-op and returns the existing handle — that's actually OK for the hook, but the shared memory / event handles from the first C# session are abandoned and `CreateFileMappingW` in the DLL was only called at load. The second C# `OpenExisting` will succeed, but there is no way to re-establish `dllReady` state or recover from a half-torn-down session.
- **Fix:** Check `GetModuleHandleW` inside the remote process (e.g. by enumerating modules via `Toolhelp32`) before injecting; skip if already present and reuse the existing shared memory. Alternatively, have the DLL detect a new client and reset state on each `dllReady` read.

### M-3 · `HomeViewModel.Poll` leaks `Process` handles every second
- **File:** `ZombieForge/ViewModels/HomeViewModel.cs:130-165`
- **Severity:** major · **Category:** Performance / Resource leak
- `Process.GetProcessesByName(n)` allocates `Process` instances which each hold an OS handle; `SelectMany(...).ToArray()` materialises them, `processes[0]` is used, and the rest are never disposed. `MainModule` also caches a `ProcessModule`. Over a long session this accumulates handles until the GC eventually finalises them. Poll runs every second.
- **Fix:** Wrap in `using`/dispose explicitly:
  ```csharp
  foreach (var p in processes) if (!ReferenceEquals(p, proc)) p.Dispose();
  try { /* use proc */ } finally { proc.Dispose(); }
  ```
  The same pattern in `MainViewModel.OnGameStartedAsync:142-144` leaks too.

### M-4 · Poll cycle reopens the process handle every second
- **File:** `ZombieForge/ViewModels/HomeViewModel.cs:165-202`
- **Severity:** major · **Category:** Performance
- `OpenGameProcess` / `CloseGameProcess` on every tick is wasteful and racy (the PID can be reused if the game restarts between ticks). Open once when the game starts, cache the handle in `HomeViewModel`, close on stop.

### M-5 · GameEventType enum is duplicated between C# and C++
- **File:** `ZombieForge/Models/GameEventType.cs:3-13`, `BlackOpsMonitor/SharedMemory.h:8-18`
- **Severity:** major · **Category:** Architecture / Correctness
- Adding a new event requires editing both files with identical numeric values or the IPC silently misinterprets. There is no build-time check.
- **Fix:** Generate one from the other (e.g. a small T4 or a pre-build script that emits the C++ header from the C# enum), or add a round-trip sanity check in `TryConnect` via a version field in `SharedGameState`.

### M-6 · `BO1ConfigService.Save` loses original line endings
- **File:** `ZombieForge/Services/BO1ConfigService.cs:95-160`
- **Severity:** major · **Category:** Correctness
- `File.ReadAllLines` discards the original line terminators; `File.WriteAllLines` writes `Environment.NewLine` (CRLF on Windows). BO1's `config.cfg` is typically CRLF so this usually works, but any user who has edited the file with LF endings will see their EOLs swapped on first save (noisy git/VCS diffs). Also `lines[i] = string.Empty` for removed binds leaves blank lines that accumulate over repeated save cycles.
- **Fix:** Read with `File.ReadAllText`, split while remembering the delimiter, and re-emit it. Remove lines via `RemoveAt(i--)` rather than blanking.

### M-7 · `ConfigViewModel.Sensitivity` persists with culture-dependent format
- **File:** `ZombieForge/ViewModels/ConfigViewModel.cs:272`
- **Severity:** major · **Category:** Correctness (partial)
- Load side correctly uses `CultureInfo.InvariantCulture` (lines 229-232) but save uses `"G"` *with* `InvariantCulture` — that's actually fine. However elsewhere `int.TryParse(fovStr, ...)` and `int.TryParse(fpsStr, ...)` are called without `NumberStyles`/`CultureInfo`; a thousands-separator locale would misparse e.g. `"1000"` — low risk but inconsistent with the sensitivity path.
- **Fix:** Pass `NumberStyles.Integer, CultureInfo.InvariantCulture` to all numeric parses of config values.

### M-8 · Fire-and-forget async without exception isolation
- **File:** `ZombieForge/ViewModels/MainViewModel.cs:117, 134, 150`; `ZombieForge/ViewModels/HomeViewModel.cs:83`
- **Severity:** major · **Category:** Error Handling
- `_ = OnGameStartedAsync();` and `_ = PollAsync(_cts.Token);` swallow any exception thrown synchronously or via the returned task. If `DllInjector.Inject` or `Process.GetProcessesByName` throws unexpectedly, the failure is invisible.
- **Fix:** Wrap with a `.ContinueWith(t => _logger.LogError(t.Exception, ...), TaskContinuationOptions.OnlyOnFaulted)` or use an `async void` handler with a top-level try/catch that logs.

### M-9 · `InitThread` sleeps 5 s before installing hook
- **File:** `BlackOpsMonitor/dllmain.cpp:72-86`
- **Severity:** major · **Category:** Correctness / Performance
- A magic `Sleep(5000)` is used to let BO1 initialise string tables. If the machine is fast and the user alt-tabs to ZombieForge, they wait 5 s for events to start; on a slow machine 5 s may still be too short and `SL_ConvertToString` can crash. The code already null-checks `scrStringGlob` in `HandleNotify`, so the delay is redundant protection.
- **Fix:** Poll `*(uintptr_t*)SCR_STRING_GLOB != 0` every 100 ms up to a timeout, then install hook. Drop the blind sleep.

### M-10 · `RemoveHook` on `DLL_PROCESS_DETACH` is unsafe
- **File:** `BlackOpsMonitor/dllmain.cpp:96-105`, `BlackOpsMonitor/Hook.cpp:169-185`
- **Severity:** major · **Category:** Correctness
- During `DLL_PROCESS_DETACH`, another thread may be executing inside `VM_Notify_Detour` / `g_trampoline`. `RemoveHook` restores the original bytes and `VirtualFree`s the trampoline without stopping the world, so a suspended thread returning to freed memory will crash the host process on exit. Also, per MSDN, calling `VirtualProtect` / `VirtualFree` from `DllMain` is discouraged.
- **Fix:** Skip `RemoveHook` during process-detach (the kernel is tearing everything down anyway), or suspend all threads first with `Toolhelp32`/`SuspendThread` and only then patch.

### M-11 · `OnGameStoppedAsync`/`OnGameStartedAsync` race
- **File:** `ZombieForge/ViewModels/MainViewModel.cs:137-170`
- **Severity:** major · **Category:** Concurrency
- If the game exits while `OnGameStartedAsync` is still on `await Task.Run(... Inject ...)`, `OnGameStopped` runs first (disposes nothing — `_eventMonitor` is still null), then `OnGameStartedAsync` completes, creates a new `GameEventMonitor`, starts it, and leaks it until the next start/stop cycle because there is no subsequent stop signal.
- **Fix:** Guard the continuation with the current `IsGameRunning` state, or track a generation counter that the continuation checks before assigning `_eventMonitor`.

### M-12 · No tests exist
- **File:** repository root
- **Severity:** major · **Category:** Testing
- Zero test projects, zero test files. `GameSession` (timing/format logic, 100 % pure) and `BO1ConfigService` (parse/serialise round-trip) are trivially unit-testable and high-value. `ConfigViewModel.ApplyDvars`/`Save` has locale-sensitive math that would benefit from coverage.
- **Fix:** Add a `ZombieForge.Tests` xUnit project for these pure components. No UI testing harness required.

### M-13 · `BO1ConfigService` parser is fragile
- **File:** `ZombieForge/Services/BO1ConfigService.cs:59-90, 162-167`
- **Severity:** major · **Category:** Correctness
- `rest.IndexOf(' ')` picks the first space, which is wrong for keys or commands containing spaces inside quotes (BO1 allows `bind X "vstr foo ; say hello"`). `Unquote` only strips one outer pair, so `"\"say \"hi\"\""` returns `say \"hi\"` with embedded quotes retained, possibly corrupting on re-save. There is no handling of comment lines (`//`) — a `seta` in a comment would be parsed as a real dvar.
- **Fix:** Use a small tokeniser that respects quoted strings, and skip lines whose first non-whitespace character is `/`.

### M-14 · `Microsoft.Extensions.Logging.Debug` referenced but unused
- **File:** `ZombieForge/ZombieForge.csproj:39`, `ZombieForge/App.xaml.cs:35-46`
- **Severity:** major · **Category:** Dependency Health
- The `Debug` extensions package is added, but `CreateLoggerFactory` does not call `AddDebug()`; Serilog's `WriteTo.Debug()` is what's actually used. The package reference is dead weight.
- **Fix:** Remove `<PackageReference Include="Microsoft.Extensions.Logging.Debug" .../>` or start using `builder.AddDebug()`.

### M-15 · `ApplicationData.Current` throws outside packaged context
- **File:** `ZombieForge/Services/LocalizationService.cs:43, 73`, `ZombieForge/ViewModels/MainViewModel.cs:184, 192`
- **Severity:** major · **Category:** Error Handling
- WinUI 3 supports unpackaged execution; `ApplicationData.Current` throws `COMException` in that mode. There is no fallback. With the current `EnableMsixTooling=true` config this is fine *today*, but a side-by-side unpackaged run (debugging from an unexpected folder, CI test host, etc.) will crash on startup before the `ResourceLoader` is even created.
- **Fix:** Wrap settings access in a try/catch that degrades to in-memory only; or gate on `Microsoft.Windows.ApplicationModel.DynamicDependency.PackageDependency` / `Package.Current.Id` presence.

## Positive Observations

- **Clean MVVM separation.** ViewModels own state, XAML uses `x:Bind` in `OneWay`/`TwoWay` as appropriate, code-behind is minimal. `HomePage.xaml.cs:25-30` correctly unsubscribes in `Unloaded` to avoid leaks.
- **Per-game abstraction is real.** Adding BO2 truly is a matter of implementing `IGameHandler`; no core service needs to change. `BO1AddressProfiles` already pre-bakes a slot for additional patches.
- **Structured logging is used consistently** (`_logger.LogInformation("State: {State}", value)`), with a sensible log-level split (Debug→VS, Warning+→file, 7-day retention) in `App.xaml.cs:29-47`.
- **Nullable reference types are enabled** project-wide and honoured — `?`/`!` usage is principled.
- **WMI-based process detection** is a nice choice vs. polling; `ProcessWatcher` correctly implements `IDisposable` and callers dispose it.
- **`DispatcherQueue.TryEnqueue` marshaling** is respected on WMI and shared-memory callbacks (`MainViewModel:104-105, 174-178`, `HomeViewModel:146-150, 185-193`), following the architecture rule.
- **`GameSession`** is a small, pure, testable state machine with clear semantics for "frozen round elapsed" vs. live counting — a nice piece of design.
- **Native hook** saves the original bytes, builds a proper trampoline, correctly identifies a 6-byte stolen prologue boundary, and wraps `SL_ConvertToString` in SEH so a bad string ID cannot crash the game.
- **Localisation scaffolding** (`LocalizationService` + `.resw`) is done correctly — initialised before window creation, persisted via `LocalSettings`, with a proper "restart required" banner.
- **Consistent file layout and naming** — folders map cleanly to the layers described in README and Copilot instructions.
